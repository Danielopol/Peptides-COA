# Peptide COA Scanner — Revised Execution Plan

## Context

The existing `peptide_coa_scanner_plan.md` is a sound MVP scaffold, and Phase 0 is already executed (81-rule extracted ruleset, scoring rubric, LLM prompt template, OCR-backed calibration on 200 originals + 1245 synthetic fakes). This revision incorporates three things the original plan didn't account for:

1. **The calibration report's own findings.** Only 5 of 81 rules carry meaningful fake-detection signal on the current corpus (FMT-003, XREF-004, META-004, META-002, FORG-007). 48 rules are no-signal; 24 are non-evaluable from a single PDF. One rule (FMT-005) is anti-discriminative.
2. **Circular validation risk in the eval set.** The 1,245 synthetic fakes were generated by `coa-faker` using perturbations the rules were written to catch. Real-world forgeries (the 5 in `FAKE/`) are dominated by Janoshik-style image reuse — a different attack class than the synthetic perturbations exercise.
3. **Monetization decisions.** User confirmed Flutter (web + Android) + accounts + freemium subscription. This affects backend scope from the start.

The goal of this revision is to (a) close the highest-leverage detection gaps the calibration explicitly recommends, (b) make the eval set actually measure real-world authenticity rather than synthetic-perturbation detection, and (c) bake the chosen client/payments architecture in early so it isn't a Phase 5 surprise.

## Phase 0.5: Calibration-Driven Rule Surgery (1–2 days)

Phase 0 is done but the calibration report flags concrete fixes that should happen before any pipeline integration. Do these now while the rules JSON is still the source of truth.

1. **Repair or remove `FMT-005` (byte-dedup clone-stamp check).** Currently anti-discriminative (orig 79% fire vs fake 43%) because originals reuse logo bytes across pages. Replace with localized pixel correlation on a sliding window, or drop entirely. Decision: drop unless replacement evaluator can be written in <2 hours.
   - File: `Rules/calibration/calibrate.py` (evaluator function), `Rules/coa_rules.json`
2. **Add `Rules/peptide_mw_table.json`** — ~50 common research peptides with expected monoisotopic mass and tolerance. Sources: PeptideRecon article + existing NUM-010 references (BPC-157 ≈1419 Da, TB-500 ≈4963 Da). New rule `XREF-009` reads claimed MW from COA and compares ±1 Da (general) or ±0.3 Da (if MS technique is MALDI-TOF) or ±0.05 Da (ESI-MS).
3. **Add `Rules/known_labs.json`** — registry of legitimate labs from the extraction report (Janoshik Analytical, Colmaric Analyticals, Novascreen Biosciences, MZ Biolabs, Freedom Analytics Labs, Vanguard Laboratory) with format fingerprints (logo hash if available, report-number pattern regex, contact info). Drives `LAB-009`.
4. **Add `XREF-010` — Janoshik verification key check.** Evaluator extracts a Janoshik task/verification key from extracted text via regex (per LAB-008 already in ruleset). At backend runtime this rule additionally **hits `janoshik.com/verify` with the key and compares returned task data to the PDF claims** (peptide name, batch, purity %). Mark this rule as critical (weight 10) since calibration report explicitly named it as the next-ceiling improvement.
   - Note: rate-limit politely, cache results, and verify their ToS before launch. If ToS forbids automated lookup, fall back to displaying the verification URL + key to the user with a "tap to verify" deeplink.

## Phase 1: Real-World Eval Set (2–3 days, can run in parallel with Phase 2)

The synthetic fakes stay for calibration breadth, but they cannot be the held-out test. Build a real-world held-out set:

1. **Seed set:** The 5 existing real fakes in `FAKE/` + 30 confirmed-real COAs sampled across all four vendor folders in `COAs/`. Annotate manually using the JSON sidecar format already in the original plan.
2. **Active collection:** Scrape Reddit `r/Peptides`, `r/PeptideSources`, `r/MoreThanPeptide` for posts tagged "fake COA" or "is this real" with images attached. Target +50 real fakes over 2 weeks. Manual triage required.
3. **Treat held-out as untouchable.** Calibration weights and prompt iterations cannot be tuned against this set. Use only for final accuracy reporting.
4. **Reporting:** Track accuracy, FPR, FNR, and score calibration separately on synthetic vs real-world. The synthetic numbers should not be quoted to users — only the real-world numbers represent real performance.

Folder layout (extends original plan):

```
Data/
├── synthetic/        (existing coa-faker output, used for breadth calibration)
├── valid/            (annotated real COAs from COAs/ subfolders)
├── eval_holdout/     (real fakes + matched real COAs, NEVER tuned against)
└── annotations/      (sidecar JSONs)
```

## Phase 2: Backend Pipeline (5–7 days)

Stack: **Python + FastAPI**, hosted on **Cloud Run or Railway** (not Vercel — PDF + OCR + Gemini exceeds 10s/50MB serverless caps). Stateless except for the database.

Endpoints:

```
POST /api/scan         — multipart upload, returns score JSON
GET  /api/history      — auth required, returns user's past scans
GET  /api/rules        — current ruleset version + summary
GET  /api/me           — auth profile + subscription status
POST /api/billing/*    — Stripe webhook endpoints
```

Processing flow inside `/api/scan`:

1. Reject non-COA inputs early: detect file type, hard-fail on >20 MB, reject if OCR returns <100 chars of text (catches blank pages / cat photos).
2. Normalize to (text, image, metadata) tuple. Use Tesseract via existing `Rules/calibration/calibrate.py` evaluator code paths for text + structural OCR. Skip Gemini Vision OCR — keep deterministic text for rule engine; pass original image to LLM.
3. Run deterministic rule engine using `Rules/calibration/coa_rules_calibrated.json` (the v1.1-calibrated file, not the raw v1.0). Add the new rules from Phase 0.5.
4. **Hard checks subroutine** (parallel to rule engine):
   - Peptide MW table lookup → XREF-009
   - Known-lab registry match → LAB-009
   - Janoshik verification API call → XREF-010
   - PDF metadata anomaly scan → META-002/META-004
5. Call Gemini Flash with image + extracted text + `Rules/llm_prompt_template.txt` + ruleset context. Use **structured output mode** (JSON schema) to force valid JSON.
6. Score aggregator: combine rule engine outputs + LLM outputs per `Rules/scoring_rubric.json`. Cap LLM influence at ~40% of final score to keep determinism.
7. Persist scan record to DB if user is authenticated; otherwise return without storing.

Critical files to create:
- `backend/app/main.py` — FastAPI entrypoint
- `backend/app/scan.py` — pipeline orchestrator
- `backend/app/rules_engine.py` — wraps existing `Rules/calibration/calibrate.py` evaluators
- `backend/app/llm_client.py` — Gemini Flash wrapper with retries + structured output
- `backend/app/checks/janoshik.py` — verification key lookup with caching + ToS-safe fallback
- `backend/app/checks/mw_table.py` — MW cross-reference
- `backend/app/scoring.py` — aggregator implementing `scoring_rubric.json`
- `backend/app/auth.py` — Firebase Admin SDK token verification middleware
- `backend/app/billing.py` — Stripe webhook handlers + Play Billing receipt validation

Reuse: the OCR + evaluator stack in `Rules/calibration/calibrate.py` is already working with cached OCR — wrap it as a callable instead of rewriting.

## Phase 3: Auth + Payments (2–3 days)

- **Auth:** Firebase Auth (email/password + Google sign-in). Flutter SDK + Firebase Admin SDK in backend. No password reset code to write, GDPR essentials handled.
- **Subscription model:** freemium — 3 free scans/calendar month, then paywall. Single SKU: monthly unlimited (price TBD; suggest $9.99/mo to start).
- **Android:** Play Billing for the subscription (Google requires it for digital goods). Use the official `in_app_purchase` Flutter package.
- **Web:** Stripe Checkout + Stripe Customer Portal for self-serve cancellation.
- **Linking:** Both Play Billing receipts and Stripe customers write to the same `subscriptions` row keyed by Firebase UID, so the user can sub on either platform and use both.
- **Free-scan accounting:** stored server-side (`users.free_scans_remaining_this_month` reset on first scan of each calendar month). Client never trusted for the count.

Critical files:
- `backend/app/billing.py` — webhook dispatcher
- `backend/app/quota.py` — free-tier counter
- `app/lib/services/auth.dart` — Firebase Auth wrapper
- `app/lib/services/billing.dart` — IAP + Stripe redirect

## Phase 4: Flutter App (7–10 days)

Build order unchanged from original plan, with two additions:
- **Auth gate before scan** (sign in / sign up screen, Google one-tap).
- **Paywall sheet** triggered when `free_scans_remaining_this_month == 0`. Shows the single sub SKU + restore purchases button.

Results screen additions:
- If `XREF-010` (Janoshik verify) returned a confirmed match, surface a green "Lab-verified ✓" badge above the score gauge. This is the highest-trust signal the app can produce.
- If `XREF-009` (MW mismatch) fires, surface as a red "Wrong molecular weight for claimed peptide" finding — this is high-confidence forgery evidence.

---

## Phase 5: Validation & Calibration (3–5 days)

Two-tier reporting:
- **Synthetic eval:** existing calibration on 200/100 sample. Use only for regression detection ("did this prompt change break anything?"). Target: maintain current discrimination on the 5 calibrated rules.
- **Real-world held-out eval:** the Phase 1 held-out set. Use for the *real* numbers reported externally. Targets stay as in original plan but quoted from this set only:
  - Accuracy > 85% (real-world)
  - FPR < 5%
  - FNR < 10%

Iteration loop: tune LLM prompt and aggregator weights against synthetic + valid set, then *report* numbers from real-world held-out. Re-run weekly until targets met.

## Verification (end-to-end)

1. **Phase 0.5 sanity:** re-run `Rules/calibration/calibrate.py` against the synthetic set after rule changes. Confirm FMT-005 no longer anti-discriminative and XREF-009 fires on a synthetic fake with edited MW (manually inject one to test).
2. **Janoshik check live test:** scan one of the `COAs/Janoshik_Tests/` PDFs through the backend; XREF-010 should return a match. Scan an `FAKE/` Janoshik-style image; XREF-010 should return a mismatch or no-key.
3. **Real-world held-out evaluation:** run the full pipeline against the held-out set, generate confusion matrix, check FPR < 5% and FNR < 10%.
4. **Quota + paywall:** create a test account, scan 3 times, confirm 4th scan hits paywall. Run Stripe test-card purchase, confirm 4th scan unlocks. Repeat for Play Billing in Android debug build.
5. **Adversarial inputs:** upload a blank PDF, a cat photo, and a non-COA PDF. All should fail early with a clear error rather than producing a misleading score.
6. **Cross-platform purchase parity:** subscribe via Stripe on web with a test account, sign in on Android emulator, confirm unlimited scans work without re-subscribing.

## Risk register

| Risk | Mitigation |
|---|---|
| Janoshik ToS forbids automated `/verify` lookups | Fallback: render verification URL + key as a deeplink the user taps. Still useful, no legal exposure. |
| Synthetic-trained model fails on real fakes | Held-out set catches this before launch. If FNR > 10%, expand real-fake collection before shipping. |
| Vendor sues over "Likely Forged" label | Add ToS + disclaimer ("indicator, not legal/medical advice"). Use softer copy: "Multiple red flags detected — verify with vendor" instead of "Forged." |
| User trusts a false-positive "Authentic" label and ingests bad peptide | Disclaimer + always surface verification-key URL so user can independently confirm. Never imply medical safety. |
| Gemini Flash latency/cost at scale | Cap scans/min, cache by file hash (same upload → same result), monitor cost per scan. Plan B: Gemma migration retained in Phase 6 only if economics demand it. |
| Account-required UX scares away top-of-funnel users | First scan can run anonymously up to result preview; account required only to *see* the breakdown. |
